`IAgenticaConfig` controls the runtime behavior of `Agentica`. The frequently used fields are:

- `selector`: chooses how candidate operations are narrowed before execution.
- `executor`: overrides orchestration steps directly.
- `systemPrompt`: customizes system prompts for internal agents.
- `retry`: controls validation feedback retries.
- `locale` and `timezone`: tell the model which user locale and time zone to assume.
- `capacity` and `eliticism`: tune the legacy LLM selector batching behavior.

`selector` is the normal configuration point for function selection. Use `executor.select` only when you need to replace the internal selection step with your own function.

Selector modes:

- `llm` or `standard`: preserves the existing full-catalog LLM selector. This is the default.
- `local`: uses the dependency-free local operation index, pushes the top candidates into the call stack, and skips the LLM selector step.
- `hybrid`: uses the local index first, then asks the LLM selector to choose from the smaller candidate list.
- `auto`: uses the default LLM selector for small catalogs and `hybrid` when the operation schema estimate reaches `autoThresholdCharacters`.

For large controller catalogs, start with `auto` or `hybrid`. `local` is useful when operation names and descriptions are explicit enough to trust lexical matching without an extra LLM selection pass.

The optional [`@agentica/vector-selector`](/docs/plugins/vector-selector) plugin is still available for semantic search backed by SQLite or PostgreSQL. Use the built-in selector first when you want zero infrastructure, and the vector selector when semantic retrieval is worth the extra embedding store.

`IAgenticaExecutor` is a lower-level escape hatch for replacing orchestration steps. For example, setting `initialize` to `null` skips the lazy initialization step and lets the agent enter selection directly.

If your goal is only to change candidate selection, prefer `config.selector`. If you need a fully custom selection algorithm, assign `executor.select` and implement it against `AgenticaContext`.

`systemPrompt` customizes the prompts used by the internal agents described in [Function Calling](/docs/concepts/function-calling/#orchestration-strategy). `common` is appended to every internal agent prompt, while the more specific properties target individual phases.

Use `common` for durable product policy, tone, and domain constraints. Use per-agent prompts when a rule applies only to selection, calling, or result description.

`retry` is the retry count for [Validation Feedback](/docs/concepts/function-calling#validation-feedback). When the model composes invalid typed arguments, Agentica reports validation errors back to the model and lets it correct the call.

Use `2` or `3` for normal production use. Set `0` only when you explicitly want to observe the first model output without validation retries.
